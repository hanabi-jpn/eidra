# Eidra Marketing Strategy

This document turns Eidra's product philosophy into a practical launch and outreach plan.

## Goal

Make Eidra feel like the default local trust layer to place in front of modern AI development workflows.

That means:

- easy to explain in one sentence
- easy to try in under 10 minutes
- easy to recommend without overselling
- easy for creators, newsletters, and media to describe correctly

## Positioning

### Category

Eidra is a local-first trust layer for AI development.

### Core promise

- See what leaves your machine
- Decide what is allowed to leave
- Prove what happened later

### What Eidra is not

- not a generic enterprise compliance dashboard
- not a hosted AI gateway that requires your data to pass through another cloud
- not an antivirus product
- not a replacement for your editor, model, or MCP server

## Ideal Users

### 1. AI-curious builders

People using Cursor, Claude Code, Copilot, or SDKs who want a safer default.

What they care about:

- "What is my AI tool sending?"
- "Can I try this without breaking my setup?"
- "Can I keep using my current tools?"

### 2. Developers building agentic workflows

People wiring MCP servers, SDKs, local models, and CI together.

What they care about:

- integration points
- machine-readable output
- policy control
- local routing and auditability

### 3. Security-minded engineering teams

People who want a lightweight boundary before broader governance tooling.

What they care about:

- local-first design
- no account and no SaaS dependency
- audit logs
- clear limits and threat model

### 4. Creators, newsletters, and media

People who need a sharp story and a fast demo.

What they care about:

- why this matters now
- who it helps
- how it is different from "another security tool"
- whether the setup friction is low enough to recommend

## Messaging Pillars

### 1. Local-first by default

Eidra does not ask users to trust another cloud layer in order to protect AI traffic.

### 2. Works with the tools people already use

Eidra is strongest when it feels additive to Cursor, Claude Code, MCP, SDKs, and CI rather than competitive with them.

### 3. Progressive adoption

People should be able to start with visibility, then add masking, blocking, routing, and gateway controls.

### 4. Built for both humans and automation

The TUI and docs help humans understand what is happening. JSON output and setup artifacts help scripts, CI, and agent tooling.

## Why Now

AI development is no longer just "send prompt, get answer."

It now spans:

- editors and coding agents
- SDK-driven applications
- MCP-connected tools
- CI and automation workflows

That means the trust problem has moved closer to the developer machine and the tool boundary.

This is the right moment to position Eidra as the local trust layer for that boundary.

## Narrative

### The old story

"AI tools are dangerous, so do not trust them."

This is too negative and does not match how developers actually behave.

### The stronger story

"AI development needs a localhost trust boundary."

That framing keeps the product:

- modern
- pragmatic
- easy to remember
- compatible with both beginner and developer audiences

## Launch Sequence

### Phase 1. Design partner outreach

Goal: get honest feedback before broad distribution.

- send warm DMs to 10 people who actually use AI coding tools
- ask for one reaction or one friction point, not a public post
- optimize setup, explanation, and first-run experience based on replies

### Phase 2. Creator and newsletter outreach

Goal: get trusted third-party validation.

- message creators who cover AI workflows, AI coding, MCP, and developer productivity
- message security-minded developers who understand prompt leakage and tool risk
- message newsletters that can summarize developer tools quickly

### Phase 3. Public launch

Goal: make the repo look immediately understandable.

- post the demo GIF
- pin the beginner explanation
- pin the developer explanation
- pin the architecture page
- link the media kit

## Outreach Rules

- ask for feedback before asking for amplification
- personalize the first line to the person's audience or recent theme
- keep the first DM under 6 lines
- send one link, not five
- offer the quickest possible path to trying it
- never lead with fear alone

## Assets Required Before Broad Outreach

- one 20-30 second demo GIF
- one static screenshot of the TUI
- one architecture diagram
- one beginner explainer
- one developer explainer
- one media kit
- one outreach playbook with reusable DM copy

## What Success Looks Like

- people understand the product in less than 30 seconds
- creators can describe it without inventing new terms
- developers can find the exact integration path they need
- media can grab a one-line summary, a 100-word summary, and a screenshot immediately

## Metrics

- DM reply rate
- "willing to try" rate
- successful setup rate
- first issue or feedback report rate
- README to docs click-through
- newsletter, tweet, or blog mentions

## Avoid These Mistakes

- do not pitch Eidra as "the most secure AI product"
- do not lead with enterprise jargon
- do not bury the quickstart below architecture details
- do not send long cold DMs
- do not ask creators to post before they understand the product

## Recommended Order Of Work

1. polish the beginner and developer docs
2. make the media kit copy usable without rewriting
3. send warm DMs
4. tighten setup friction
5. expand to creators and newsletters
6. do a public launch after the first feedback loop
